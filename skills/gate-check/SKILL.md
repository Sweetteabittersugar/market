# Gate Check — 阶段门控检查

> 每个阶段结束前执行。加载 `docs/templates/gate-checklist.md`，结构化检查，输出判定。

## 何时用

- 任何阶段完成后，用户说 /gate-check、门控检查
- 三次确认门控的正式执行
- 用户不确定能否进入下一阶段时

## 流程

### 1. 加载清单
读 `docs/templates/gate-checklist.md`，确认当前阶段的特定检查项。

### 2. 选 3 个维度
从 4 个通用维度中选最相关的 3 个：
- **正确性** — 产出是否对、数据流是否断裂
- **安全性** — 密钥/注入/校验
- **健壮性** — 错误处理/日志/回退
- **可维护性** — 命名/注释/文件大小

### 3. 逐维度检查
每维度：
1. 逐条检查（grep/读文件/跑测试——不是猜）
2. 输出该维度的 [x] [!] [ ] 清单
3. 给出维度结论：PASS / CONCERNS / FAIL
4. **等用户确认后再下一维度**（三次确认门控的核心）

### 4. 输出总结
- 总体判定：PASS / CONCERNS / FAIL
- 阻塞问题清单（FAIL 项）
- 改进建议清单（CONCERNS 项）

## 规则

- 不猜测——每个检查项用实际数据（grep 结果、测试输出、文件内容）
- FAIL 必须给出具体文件:行号 + 修复建议
- CONCERNS 不阻塞，但必须记录，让用户决定
- 用户说"够了进下一步"→ 直接跳过，不纠缠
- 三次确认不是 AI 三个角度自由发挥——是**三个维度逐项检查**

## 输出格式

```
## 门控报告 — Stage {N}: {名称}

### 维度 1: {名称}
- [x] 检查项: 通过 — 证据: {grep结果/测试输出}
- [!] 检查项: 有问题 — 文件:行号 — 建议: ...
- [ ] 检查项: 未检查

结论: PASS / CONCERNS / FAIL

### 维度 2: {名称}
...

### 维度 3: {名称}
...

### 总体判定: PASS / CONCERNS / FAIL

### 阻塞问题
- [ ] {严重度}: 问题 — 位置 — 修复建议

### 改进建议
- {建议}
```
