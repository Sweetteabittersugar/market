# 全栈开发工作流 v7

> 2026-06-20 | 10 阶段 + 故事分解 + 角色分工 + 门控清单 + 复盘。借鉴 BMAD method 故事驱动模式。

## 核心理念

**AI 编程失败不是因为模型不够强——是因为意图、上下文和验证没对齐。**

## 全局规则

1. **Commit every session**：`git add -p && git commit -m "WIP: checkpoint"`。`git reset --hard` 是终极撤回。
2. **一次一个决策**：不让 AI 一次做三件事。
3. **AI 自审**：写完代码后让它自己审查。
4. **审查用不同模型**：同家族审查 = 盲点叠加。
5. **Walking Skeleton 优先**：先一条垂直切片跑通，再横向扩展。
6. **三次确认门控**：阶段结束时 Agent 查 `docs/templates/gate-checklist.md`，从 4 维度选 3 个最相关的逐维度确认。
7. **调研先于决策**：涉及技术选型/定价/最佳实践/合规 → 先 `/research`，不凭训练数据做决策。

## 阶段门控：三次确认 + 结构化清单

门控清单：`docs/templates/gate-checklist.md`。4 个通用维度：
1. **正确性** — 是否符合 Spec，数据流有无断裂
2. **安全性** — 密钥/注入/校验
3. **健壮性** — 错误处理/日志/回退
4. **可维护性** — 命名/注释/文件大小

```
阶段完成 → Agent 加载 gate-checklist.md → 选 3 个最相关维度 → 逐维度检查
         → 用户确认 → 下一维度 → 用户确认 → 第三维度 → 用户确认 ✅ → 下一阶段
         → 用户说"够了进下一步" → 普通项跳过
         → 🔴 硬门控不可跳过，必须 PASS
中途"修" → 归零。判定: PASS / CONCERNS / FAIL。
```

#### 层级式（Hierarchical）— 执行阶段用

> 主 Agent 掌舵，辅助 Agent 并行侦察。主 Agent 说了算。

```
阶段开始 → 人确认范围
         → 主 Agent 启动（拥有最终输出权，可写文件）
         → 辅助 Agent 并行启动（独立上下文，只分析/验证）
         → 辅助结果汇入主 Agent
         → 主 Agent 综合，裁决冲突，产出交付物
```

| 角色 | 职责 | 权限 |
|------|------|------|
| 主 Agent | 拥有最终输出、综合判断、产出交付物、写 Dev Record | 读写文件 |
| 辅助 Agent | 独立视角、单一维度、输出建议 | 只读 |

```
# 阶段 4（后端实现）— 层级式示例：
主:    coder              → 写代码
辅1:   tdd-guide          → 同时写测试（红灯）
辅2:   python-reviewer    → 代码落盘后立即审查
辅3:   test-generator     → 同时生成边界测试

# ❌ 错误：等 tdd-guide 写完测试再让 coder 写 — 串行浪费
# ✅ 正确：同时启动，coder 边写边看辅助结果
```

---

#### 对话式（Collaborative）— 规划阶段用

> 无主辅之分。多 Agent 平等讨论，碰撞出最优解。人做最终决策。

```
阶段开始 → 人抛出问题
         → 3-5 个 Agent 同时启动（不同角色/视角）
         → 各 Agent 独立输出分析
         → 人（或指定协调 Agent）综合各方观点
         → 形成共识或记录分歧
         → 协调 Agent 产出最终交付物
```

| 角色 | 职责 |
|------|------|
| 协调 Agent | 综合各方观点、记录分歧、产出最终文档（不裁决，只总结） |
| 参与 Agent | 独立分析、提出观点、挑战他人假设 |

```
# 阶段 0（需求分析）— 对话式示例：
ceo:                  → "这个需求商业上合理吗？有没有更简单方案？"
pm:                   → 用户旅程 / 异常边界 / 约束清单
critic:               → "这个需求的前提假设成立吗？"
knowledge-synthesizer → 跨项目经验："上次类似需求踩过 X 坑"
人:                   → 综合各方 → 最终决策
```

**关键差异**：对话式不设主 Agent——所有 Agent 平等发言，人做最终裁判。因为规划阶段没有"正确答案"，需要碰撞。

---

#### 流水线式（Pipeline）— 交接阶段用

> A→B→C 串行。每段产出通过 Story 文件/Handoff 包传递。下游不重复上游工作。

```
阶段N 完成 → Handoff 包（Story Dev Record / sprint.yaml / NOW.md / decisions/）
          → /compact
          → 阶段N+1 启动 → 读 Handoff 包 → 基于上游产出继续
```

核心机制：**下游 Agent 不重新分析——信任上游决策。** 发现问题 → 回退协议，不静默修正。

```
# 阶段 3→3.5 示例（API→故事分解）：
api-designer 产出 API 契约
  → /create-story 读契约 → 拆 Story → 生成 Story 文件
  → Story 文件 Context 段引用 API 契约链接
  → coder 打开 Story 文件 → 契约已在 Context 段 → 直接实现

# 如果 coder 发现 API 设计有问题：
# → 不直接改 API → 触发回退协议 → 回到阶段 3 修复
```

#### 冲突裁决

| 场景 | 层级式 | 对话式 | 流水线式 |
|------|--------|--------|---------|
| Agent 意见矛盾 | 主 Agent 裁决 | 人裁决，记录分歧 | 上游优先，或触发回退 |
| 发现阻塞问题 | 主 Agent 暂停，报告 | 协调 Agent 标记，继续 | 暂停，报告用户 |
| Agent 结果为空/出错 | 返回 null，主 Agent 继续 | 忽略该 Agent，继续 | 阻塞下游，需解决 |
| 用户推翻决策 | 用户说了算，写入 Dev Record | 同左 | 同左 |

## 阶段间协议

> 借鉴 Scrum 的 DoR/DoD 双门控 + BMAD 的 Story 文件传递链。

### 双门控：DoR + DoD

```
DoR（Definition of Ready）  →  入口条件。满足才进阶段。
DoD（Definition of Done）   →  出口条件。满足才算阶段完成。
```

| | DoR（入口） | DoD（出口） |
|---|-----------|-----------|
| **问** | "可以开始了吗？" | "真的做完了吗？" |
| **来源** | Scrum INVEST + BMAD story validation | Scrum DoD + 门控清单 |
| **检查时机** | 阶段开始前 | 阶段结束时（三次确认） |
| **谁检查** | 主 Agent 读 Handoff 包自检 | 主 Agent + 辅助 Agent 交叉检查 |

### 通用 DoR（所有阶段入口条件）

阶段 N 开始前，主 Agent 自检：

- [ ] 上游 Handoff 包完整（Story Dev Record / sprint.yaml / NOW.md / decisions）
- [ ] 上游门控通过（PASS or CONCERNS 已确认）
- [ ] 本阶段输入物存在且可读（Spec / API 契约 / Story 文件）
- [ ] 依赖的 Story 已完成（sprint.yaml 中状态为 done）
- [ ] 人已确认范围

### 通用 DoD（所有阶段出口条件）

阶段 N 结束时，主 Agent 自检 + 三次确认：

**代码/交付物质量：**
- [ ] 阶段交付物完整（Spec / 代码 / 测试 / 文档）
- [ ] 代码已自审（AI 先审自己写的）
- [ ] 命名/注释/文件大小符合规范

**测试：**
- [ ] 单元测试通过（如有）
- [ ] 验收标准逐条验证通过
- [ ] curl/Playwright 能跑通关键路径

**交接准备：**
- [ ] Story 文件 Dev/QA Record 已填写
- [ ] sprint.yaml 状态已更新
- [ ] NOW.md 精确值已写入
- [ ] 重要决策写入 decisions/
- [ ] 代码已 commit（含 Story 文件 + sprint.yaml）

**安全（如涉及）：**
- [ ] grep 密钥无硬编码
- [ ] 输入校验完整
- [ ] 错误消息不泄露内部信息

### 阶段特定 DoR/DoD

| 阶段 | 特定 DoR | 特定 DoD |
|------|---------|---------|
| 0 需求 | 有项目想法/痛点描述 | 阶段适用清单已输出，反派挑战已通过 |
| 1 Spec | 需求分析通过 | 功能清单 v1/v2 边界明确，AC 可验证 |
| 2 技术选型 | Spec 已定稿 | 三问已答，核心风险已验证，致命问题已排除 |
| 3 API | 技术栈已确定 | 每个端点有请求/响应/错误格式，契约双方确认 |
| 3.5 故事 | API 契约已定 | 每 Story ≤2 文件，Context 段完整，依赖无循环 |
| 4/5 实现 | Story Context 段完整 | 每端点 curl 通，三态覆盖，Dev Record 已填 |
| 6 验证 | 实现完成，Dev Record 有内容 | 门控 PASS/CONCERNS，QA Record 已填 |
| 7 视觉 | 前端实现完成，页面可渲染 | 截图通过人工确认，≤5 轮，a11y 检查通过 |
| 8 部署 | QA Record 通过 | 回滚已验证，冒烟通过 ≥15min，安全扫描无高危 |
| 9 复盘 | sprint.yaml 本轮全 done | 三问已答，教训已写入 memory |

### 上下文传递链（BMAD Story 文件模式）

```
阶段 N 完成
  ├── Story 文件更新（Dev/QA Record → 下游知道做了什么/有什么坑）
  ├── sprint.yaml 状态变更（机器可读）
  ├── NOW.md 焦点更新（精确值：端口/阈值/版本）
  ├── decisions/ 架构决策（如有，防同一决策反复讨论）
  ├── Commit（代码+Story+sprint.yaml+decisions）
  └── /compact
       ↓
阶段 N+1 启动
  ├── 读 sprint.yaml（做到哪了）
  ├── 读 Story 文件（完整上下文，不翻架构文档）
  ├── 读 NOW.md（当前焦点+阻塞+精确值）
  ├── 读 decisions/（相关架构决策）
  ├── 主 Agent DoR 自检
  ├── 人确认范围
  └── 开始干活
```

不需要人重复解释项目背景——全在文件里。

### 新会话启动流程

```
会话开始
  → 读 NOW.md（做到哪了+阻塞+精确值）
  → 读 sprint.yaml（当前迭代 + Story 状态）
  → 读当前 Story 文件（获取完整上下文）
  → 读 decisions/ 相关决策
  → DoR 自检通过 or 标记缺失
  → 人确认范围
  → 开始干活
```

## 12 阶段

### -1. 灵感捕捉
> 适用：所有新项目 | 协作模式: 对话式

把模糊想法变成可验证的问题表述。不写代码。

- 问题表述 → AI 追问假设和边界 → 竞品调研（≥3 个）→ 选方向

### 0. 需求分析
> 适用：所有新项目 | Agent: pm | 协作模式: 对话式

了解全貌 + **挑战需求本身**。

- 四维度：用户旅程 / 异常边界 / 约束清单 / 干系人
- 故障状态地图（≥5 个异常场景）
- AI 追问盲区
- **挑战需求**：AI 用反派角色质疑——"这个需求真的对吗？有没有更简单的方法？用户真正的痛点是什么？"
- **阶段适用清单**：结束时输出，明确哪些阶段适用/跳过，写入 handoff

```
阶段适用清单：
  -1 灵感        [✅/❌]
   0 需求        [✅/❌]
   1 Spec        [✅/❌]
   ...
```
不适用就标记 ❌，后续阶段自动跳过。

### 1. Spec
> 适用：所有新项目 / 大重构 | 协作模式: 层级式

写出 AI 能精确执行的规格书。

- 项目宪章（做什么/不做什么/约束）
- 功能清单（v1 做/v1 不做/v2 以后）
- 验收标准（可验证，不是"用户体验好"）

### 2. 技术选型 + 可行性分析
> 适用：需要新技术决策的项目 | 协作模式: 对话式

- 三问：有现成的吗？一定要自己写吗？选错了怎么办？
- 可行性：核心技术风险？最大风险怎么验证？最小可行边界？有没有致命问题？

### 3. 数据/API 设计
> 适用：有后端/数据层的项目 | 协作模式: 层级式

Contract-First。API 是前后端之间的法律合同。

- 数据建模 → ASCII 表 → 人确认
- API 端点 + 错误格式
- 迁移策略：Expand-Contract

### 3.5. 故事分解 ← 新增
> 适用：有 ≥3 个 Story 的项目（单 Story 直接跳过）| Agent: pm | 协作模式: 层级式

**借鉴 BMAD：把 Spec 和 API 拆成 Agent 能独立实现的 Story 文件。** 每个 Story 文件包含完整上下文，agent 打开就能干活，不用翻架构文档。

- 从 Spec 功能清单提取 Epic 列表（≤3 个）
- 每个 Epic 拆 1-5 个 Story，每个 Story ≤2 文件变更
- 用 `docs/templates/story-template.md` 模板生成 `docs/stories/S{XX}-{slug}.md`
- Context 段必须填满——Spec 链接、API 契约、设计决策、参考代码
- 标注依赖关系（无循环）
- 更新 `.context/sprint.yaml`
- 使用 `/create-story` 技能或手动填模板

跳过条件：项目只有 1 个 Story（改一个文件就完事）。

### 4. 后端实现
> 适用：有后端的项目 | Agent: coder | 协作模式: 层级式

每个端点 curl 能调通。

- 逐 Story：读 `docs/stories/S{XX}.md` → 测试 → 红灯 → 实现 → 绿灯 → curl → 自审
- 人审：错误处理 / 输入校验 / 日志

### 5. 前端实现
> 适用：有 UI 的项目（纯 CLI/对话项目跳过）| Agent: coder | 协作模式: 层级式

页面能渲染，能调通后端。

- 组件树 → 叶子到根
- 每组件：写 → build → 自审 → 截图
- 三态：loading / empty / error

### 6. 功能验证
> 适用：前后端都有的项目（纯后端/CLI 跳过前端部分）| Agent: qa | 协作模式: 层级式

核心用户流程跑通。功能正确性确认。

- Playwright 5-10 条关键路径
- 契约验证（前后端类型一致）
- 门控检查（按 gate-checklist.md 4 维度）

### 7. 视觉打磨
> 适用：有 UI 的项目 | 协作模式: 层级式

从"能跑"到"好看"。

- 机械检查（颜色/间距/对比度/响应式）→ 截图 → 人判断
- 3-5 轮上限

### 8. 预发布检查 + 部署
> 适用：所有要上线的项目 | Agent: qa | 协作模式: 层级式（含 🔴 硬门控）

- **安全扫描**：grep 密钥/硬编码 → 依赖漏洞 → 注入风险
- 回归测试全绿
- 回滚命令写好并验证过
- 部署 → 冒烟测试 → 15 分钟观察

### 9. 复盘 ← 新增
> 适用：所有项目（哪怕只花 5 分钟）| 协作模式: 层级式

**不跳过这一步。** 否则同样的坑踩两遍。

三问：
1. **什么做对了？**（保留、强化）
2. **什么造成摩擦？**（返工轮次、人推翻 AI 决策、AI 跑偏）
3. **下次怎么改？**（写到 rules/lessons 里，下次自动加载）

产出：更新 `~/.claude/lessons.md` 或项目 `MEMORY.md`。

## 防退化

| 现象 | 应对 |
|------|------|
| AI 反复修同一个 bug | 停。新开会话 |
| 一个功能超 5 轮没对 | 回 Spec 确认范围 |
| AI 一次生成超 300 行 | 拆小批次 |
| AI 冲下一阶段 | 三次确认归零 |
| 同类型错误犯两次 | 复盘没做 → 阶段 9 |
| Agent 丢失上下文 | Story 文件 Context 段没填满 |
| AI 凭记忆回答技术问题 | `/research` 验证后再答 |

## 回退协议

**上游决策错了不是灾难。装死才是。**

1. 发现上游阶段决策有误 → **立即停**，不继续往下写
2. 在 NOW.md 记录：什么决策错了、为什么现在才发现、影响范围
3. **回到出错阶段**重新走（不是全流程重来——已通过的阶段保留）
4. 回退后重新走的阶段**必须用不同模型审查**（防同模型盲点）
