# Spec — 需求→Spec 流水线

> 覆盖阶段 -1/0/1：把模糊想法变成 AI 能精确执行的规格书。

## 何时用

- 新项目启动，用户说 /spec、写 Spec、生成规格书
- 大重构前，需要重新理清需求
- 阶段 0 需求分析完成后，进入阶段 1

## 流程

### 第一步：灵感捕捉（Stage -1）
1. 让用户用一句话描述想法
2. 追问 3 个问题：谁用？核心痛点是什么？现有方案为什么不满意？
3. 竞品调研：搜 ≥3 个同类方案，输出差异化定位表
4. 选方向，输出一句话问题陈述

### 第二步：需求分析（Stage 0）
1. **用户旅程**：核心用户的一天，从触发到完成
2. **异常边界**：≥5 个故障场景及其应对思路
3. **约束清单**：技术/时间/人力/预算硬约束
4. **反派挑战**：AI 用反派角色质疑——"这个需求真的对吗？有没有更简单的方法？"
5. 输出**阶段适用清单**：哪些阶段适用/跳过

### 第三步：Spec 生成（Stage 1）
1. **项目宪章**：做什么 / 不做什么 / 技术约束
2. **功能清单**：v1 做 / v1 不做 / v2 以后
3. **验收标准**：每个功能 1-3 条 Given/When/Then
4. 写入 `docs/{project}-spec.md`

## 规则

- Spec 必须可验证——不写"用户体验好"，写"用户能在 3 秒内完成登录"
- 功能清单明确边界——"不做"和"做"一样重要
- 阶段适用清单诚实——不适用就标记 ❌，别硬走
- 输出到 `docs/` 目录，文件名用项目名+spec

## 输出

- `docs/{project}-spec.md`：完整规格书
- 阶段适用清单（内嵌在 Spec 中）
- 一句话总结给 boss
