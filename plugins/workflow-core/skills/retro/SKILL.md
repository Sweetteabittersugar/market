# Retro — 复盘教训提取

> 阶段 9。不跳过。扫描改动 → 识别模式 → 写入记忆。

## 何时用

- 阶段 9 复盘，用户说 /retro、复盘、总结教训
- 大任务结束后想提取经验
- 同类型错误出现 ≥2 次时

## 流程

### 1. 扫描范围
- 默认：最近 10 个 commit 或当前迭代的所有 commit
- 用户指定范围：`/retro v1.0..v1.1` 或 `/retro --last 20`

### 2. 提取信息
- `git diff --stat`：哪些文件变了，规模多大
- `git log --oneline`：commit 历史
- 读变更文件：识别模式

### 3. 四维度扫描

#### 代码规范退化
- 文件大小是否在增长（>400 行？>800 行？）
- 命名是否一致（snake_case / camelCase）
- 是否有调试代码残留（print/console.log）

#### 重复代码
- 同一逻辑出现 ≥2 次？
- 是否应该抽象但被复制了？

#### 接口设计
- API 契约是否被绕过？
- 前后端类型是否一致？
- 错误格式是否统一？

#### 安全苗头
- 是否有接近硬编码的密钥？
- 输入校验是否完整？
- 错误消息是否泄露内部信息？

### 4. 复盘三问
1. **什么做对了？**（保留、强化、写成模式）
2. **什么造成摩擦？**（返工轮次、人推翻 AI 决策、AI 跑偏、上下文丢失）
3. **下次怎么改？**（具体可操作，不是"下次注意"）

### 5. 写入记忆
- 教训 → `~/.agency/memory/episodic/active/`（跨项目自动注入）
- 项目级经验 → `D:\ai\MEMORY.md`（如不存在则建）
- 同类问题 ≥3 处 → 建议加门控
- 被纠正 ≥2 次 → 建议写规则

## 规则

- 不跳过。哪怕只花 5 分钟。
- 每条教训必须有来源（commit hash / 文件名:行号）
- 用"为什么发生"解释根因，不只用"发生了什么"
- 建议必须是可操作的——不写"以后注意"，写"在 PreToolUse hook 加 XX 检查"

## 输出

- 复盘报告（Markdown，可直接给 boss 看）
- 写入的 memory 文件列表
- 建议的规则/门控变更
